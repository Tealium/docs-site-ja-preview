An Introduction to the Data Layer

この記事では、データレイヤーについて紹介し、それが何であるか、どのように機能するか、および、実装するメリットを説明します。

データレイヤーとは

データレイヤーは、デジタル プロパティからフローするカスタマー インタラクション トラッキングの仕様です。こうしたトラッキング対象のアクティビティは、多くのソース (デスクトップやモバイル ウェブサイト、モバイル アプリ、コネクテッド デバイス、オフライン ソースなど) から生じます。データレイヤーは、サードパーティ ベンダー ソリューションを起動させ、データ駆動型イニシアチブの基盤となります。

データレイヤーを定義するには、まず、訪問者の最も重要なアクティビティ、いわゆる _イベント_を特定します。イベントには、ページ ビューやサイト検索から、ログイン、購入までのあらゆるものや、重要なビジネス決定の情報を提供するカスタマー アクションが含まれます。各イベントには、コンテキスト上の情報、いわゆる _イベント属性_が含まれます。データレイヤーは、こうしたイベントと、それに対応する属性について、そのリストで構成されています。

Tealium データレイヤーでは、トラッキングを行うイベントを特定するために tealium_event という変数を用いています。独自のイベント名を定義することができますが、作業を開始するうえでの 標準的な一連のイベント を用意しています。

多くのデータレイヤー定義に共通するイベントの例は、以下のとおりです。

例

サイト検索

イベント属性 説明 値の例
tealium_event トラッキングされているイベント。 search
page_type 表示されているページのタイプ。 search
search_keyword 検索に用いる用語。 red boots
search_results 返された検索結果の数。 42

ニュースレターの登録

イベント属性 説明 値の例
tealium_event トラッキングされているイベント。 newsletter_signup
newsletter_name ニュースレターの名称。 Weekly Digest

カートへの追加

イベント属性 説明 値の例
tealium_event トラッキングされているイベント。 cart_add
product_id カートに追加された商品の ID 。 B51378
product_name カートに追加された商品名。 Widget XYZ
product_quantity カートに追加された商品の数量。 2

ビジネスについて、こうしたイベントと属性データレイヤーが定義されれば、データレイヤーは、すべてのデジタル資産と顧客インタラクション全体にわたって唯一の真実のデータを定義する機能を果たします。

データレイヤーが必要な理由

データレイヤーは、最も価値のある資産であるデータの所有権 (オーナーシップ) を把握するのに役立ちます。ビジネスを起動させるために多くのサードパーティベンダーに依存しており、各ベンダーは、独自のデータ収集の必要性を有していますが、データレイヤーは、こうしたデータの集中型の定義となります。データレイヤーが十分に定義され管理されれば、以下のメリットがもたらされます。

  • ガバナンス データレイヤーに関するガバナンス ポリシーを制定して、データを統制できます。
  • 標準化 組織で理解しやすい用語を使用できます。 
  • マーケティングの俊敏性 データレイヤーを十分に実装すれば、マーケティング戦略について、開発リソースへの依存をなくして、変化する状況に応じた迅速な調整が可能になります。 
  • 時間と費用の節約 デベロッパーにかかる時間と費用を節約するには、ウェブサイトとモバイル アプリに対する漸進的な変更の継続なリクエストをなくします。

仕組み

データレイヤーを定義した後、デジタル プロパティ全体にこれをインストールします。データレイヤーの仕様が一貫して準拠されるよう確保するために各プラットフォーム用のインストール ガイドに従います。そうすれば、こうしたデータがタグ ベンダー構成やキャンペーン戦略で使用できます。

シンプルで理解しやすい名称設定方法を推奨します。以下に例をいくつか示します。

イベント属性 説明 値の例
order_id 完了した注文の一意 ID。 1234
order_total 注文に支払われた総額。 123.45
customer_id 訪問者の一意 ID。 0123456789
customer_zip 訪問者の郵便番号。 92101

こうした属性は、サードパーティ ベンダーに対してそれらが必要とする形式で送付可能です。Tealium Customer Data Hub を用いることで、このプロセスを簡素化できます。

たとえば、 order_id という名前の属性は、ベンダーが参照しますが、その際、 s.purchase_id、 transid、 pid、 oid などのさまざまな異なる名前を用いています。データレイヤーが order_id をオンライン購入として定義すれば、データの単一ソースを再使用して、他のベンダーに対してそれらが想定する形式でこの値を送信できるようになります。

イベント属性 イベント値 ベンダー属性 ベンダー名
order_id 1234 s.purchase_id Adobe Analytics
transid Rocket Fuel
 pid Twitter Conversions
oid Conversant
